# Easy PRD: 공유 메일 정리와 홍보 문안 관리

## 문서 범위

이 문서는 공유 메일을 수신해 Codex CLI로 분류하고 홍보 문안 초안을 생성한 뒤, 담당자가 원본 메일과 결과를 비교·검토하는 서비스의 첫 프로토타입을 정의한다.

현재 산출물은 제품 요구사항 명세이며 실제 개발 결과물이 아니다.

## 문제와 핵심 사용자

- **문제:** 담당자가 공유 메일을 직접 읽고 분류·요약·홍보 문안 작성까지 반복해야 하며, 마감일이나 모집 대상 같은 핵심 정보가 누락될 수 있다.
- **핵심 사용자:** 대학 부서의 공유 메일을 관리하고 외부 프로그램·교육·채용 정보를 홍보하는 담당자.

## MVP 핵심 기능

1. 강남대학교 IMAP 계정으로 로그인하고 unread 메일을 수동 동기화한다.
2. 메일 원문을 저장하고 유형, 대상, 마감일, 혜택, 신청 방법, 문의처를 AI로 추출한다.
3. 홍보 문안 초안을 생성하고, 중복 메일은 다시 가져오지 않는다.
4. 담당자가 원본 메일과 AI 분석 결과를 비교·검토한다.

## 프로토타입 화면 흐름

1. **로그인:** 포털 ID와 비밀번호를 입력해 IMAP에 로그인한다.
2. **메일함:** `메일 동기화` 버튼으로 unread 메일을 가져온다.
3. **검토 목록:** 가져온 메일과 처리 상태를 확인한다.
4. **상세 검토:** 원본 메일, 분류·추출 결과, 홍보 문안 초안을 한 화면에서 비교한다.

## 1. 선택한 부서 문제 브리프

- **문제 브리프:** 공유 메일의 정보를 홍보 문안으로 옮기는 동안 마감일, 모집 대상, 혜택 같은 핵심 정보가 누락되거나 처리가 늦어질 수 있다.
- **연결된 부서 제안:**
    - 이메일 통합 관리 시스템
    - 공지사항·홍보문 자동 생성기
    - 주제: `공유 메일/홍보: 공용 메일 정리와 홍보 문안 관리`

## 2. 주요 사용자와 상황

- **주요 사용자:** 대학 부서의 공유 메일함을 확인하고 외부 프로그램·채용·교육 정보를 홍보하는 담당자
- **서비스를 사용하는 순간:** 공유 메일함으로 들어온 메일을 분류하고, 홍보에 필요한 핵심 정보를 추려 문안을 다시 작성할 때

## 3. 문제와 근거

- **사용자가 반복하는 행동:**
    1. 수신 메일을 열어 내용을 읽는다.
    2. 채용·교육·대외활동 등 메일 유형을 판단한다.
    3. 모집 대상, 마감일, 혜택, 신청 방법을 찾는다.
    4. 홍보에 필요한 문장으로 다시 작성한다.
    5. 원본과 홍보 문안을 비교하며 사실을 확인한다.
- **현재 해결 방식:** 담당자가 메일을 직접 읽고 분류한 뒤, 필요한 내용을 복사하거나 다시 작성한다.
- **가장 큰 불편:**
    - 메일마다 같은 정보를 반복해서 찾아야 한다.
    - 마감일이나 모집 대상 같은 중요 정보가 누락될 수 있다.
    - 처리해야 할 메일이 많으면 홍보가 늦어질 수 있다.
    - 담당자가 바뀌면 분류 및 문안 작성 기준이 달라질 수 있다.
- **부서 설명에서 확인한 사실:**
    - 공유 메일함으로 채용공고, 직업훈련, 대외활동, 외부 프로그램 정보가 들어온다.
    - 담당자는 메일에서 유형, 대상, 마감일, 주요 혜택을 확인한다.
    - 메일 정보를 홍보 문안으로 다시 작성하는 업무가 있다.
    - 최종 내용은 담당자의 검토가 필요하다.
- **팀이 가정한 내용:**
    - Codex CLI가 메일 본문을 읽을 수 있는 형식으로 전달받을 수 있다.
    - 메일 유형과 핵심 정보를 일정한 출력 형식으로 정리할 수 있다.
    - 홍보 문안 초안이 담당자의 반복 작성 시간을 줄여줄 수 있다.
    - 첫 프로토타입에서는 운영용 메일 서버가 아닌 테스트용 SMTP 환경으로 전체 흐름을 검증할 수 있다.
    - 실제 부서의 분류 기준과 홍보 문안 양식은 아직 확인되지 않았다.

## 4. Before–Using–After

- **Before:** 담당자가 많은 메일을 하나씩 열어 유형과 핵심 정보를 찾고 홍보 문안을 직접 작성한다.
- **Using:** 테스트 SMTP 서버가 가상 메일을 수신하고, 해당 내용을 Codex CLI에 전달한다. Codex CLI는 메일을 분류하고 핵심 정보를 추출한 후 홍보 문안 초안을 생성한다.
- **After:** 담당자는 한 화면에서 원본 메일, 분류 결과, 핵심 정보와 홍보 초안을 비교하고 검토한 뒤 홍보 업무에 사용한다.

## 5. How Might We 질문

> 어떻게 하면 공유 메일 담당자가 원본 메일의 중요 정보를 놓치지 않으면서, 분류와 홍보 문안 작성에 쓰는 반복 작업을 줄일 수 있을까?

## 6. 비교한 해결책과 선택 이유

아래 비교안은 현재 제공된 내용을 바탕으로 정리한 **팀 검토용 가정**이며, 실제 팀 비교 결과로 확정된 사실은 아니다.

| 후보                   | 방식                                            | 장점                                              | 한계                                            |
| ---------------------- | ----------------------------------------------- | ------------------------------------------------- | ----------------------------------------------- |
| 수동 복사·붙여넣기     | 담당자가 메일을 Codex CLI에 직접 입력           | 구현이 가장 단순함                                | 메일을 옮기는 반복 작업이 남음                  |
| 메일 전달 전용 주소    | 기존 메일을 별도 주소로 전달                    | 기존 메일 환경을 유지하기 쉬움                    | 담당자의 전달 작업 또는 자동 전달 설정이 필요함 |
| 테스트 SMTP 연동       | 서버가 메일을 받아 Codex CLI 처리 흐름으로 전달 | 수신부터 결과 생성까지 전체 흐름을 검증할 수 있음 | 서버 구축과 메일 파싱이 필요함                  |
| AI 없는 규칙 기반 처리 | 키워드와 정규식으로 분류·추출                   | 결과 기준이 명확하고 비용이 낮음                  | 다양한 메일 형식과 자연어 문안 작성에 취약함    |

- **비교한 후보:** 수동 입력, 메일 전달 전용 주소, 테스트 SMTP 연동, AI 없는 규칙 기반 처리
- **선택한 해결책:** 테스트 SMTP 서버와 Codex CLI를 연결한 메일 분류·홍보 문안 생성 프로토타입
- **선택 이유:**
    - 사용자가 제안한 전체 흐름을 직접 보여줄 수 있다.
    - 단순 문안 생성뿐 아니라 메일 수신부터 담당자 검토까지 검증할 수 있다.
    - 입력, AI 처리 결과, 담당자의 다음 행동이 분명하다.
- **AI가 없는 대안:** 제목·본문의 키워드와 날짜 패턴을 이용해 메일을 분류하고, 추출된 정보를 고정 홍보 템플릿에 삽입한다.

## 7. 첫 프로토타입의 핵심 가치

수신 메일을 자동으로 분류하고 홍보 문안의 검토 가능한 초안을 생성하여, 담당자가 **처음부터 다시 작성하지 않고 사실 확인과 수정에 집중하도록 하는 것**이다.

AI 결과는 최종 홍보물이 아니며, 담당자의 검토를 거쳐야 한다.

## 8. 첫 프로토타입에서 보여줄 기능

1. 테스트 SMTP 서버를 통한 가상 메일 수신
2. 수신 메일의 제목·본문·발신자 정보 파싱
3. 메일 내용을 Codex CLI 처리 과정으로 전달
4. 메일 유형 분류
5. 다음 핵심 정보 추출
    - 모집 또는 참여 대상
    - 신청 마감일
    - 주요 혜택
    - 신청 방법
    - 문의 또는 참고 정보
6. 홍보 문안 초안 생성
7. 원본 메일과 다음 결과를 함께 표시
    - 분류 결과
    - 추출된 핵심 정보
    - 홍보 문안 초안
8. 담당자가 결과를 읽고 검토할 수 있는 흐름

### 전체 기능과 프로토타입 범위

| 구분        | 전체 서비스                          | 첫 프로토타입                        |
| ----------- | ------------------------------------ | ------------------------------------ |
| 메일 수신   | 실제 부서 메일을 안정적으로 수신     | 가상 메일을 테스트 SMTP로 수신       |
| 분류        | 실제 부서 기준에 따른 다양한 분류    | 제한된 예시 유형 분류                |
| 정보 추출   | 여러 메일 형식 지원                  | 준비된 가상 메일 형식 중심           |
| 문안 생성   | 부서의 실제 홍보 양식 적용           | 검토 가능한 일반 홍보 문안 생성      |
| 담당자 처리 | 실제 업무에 사용할 수 있는 운영 흐름 | 원본과 결과를 비교하는 시연 흐름     |
| 보안·운영   | 인증, 권한, 로그, 개인정보 보호 필요 | 실제 개인정보와 운영 계정 사용 안 함 |

## 9. 이번에 구현하지 않을 기능

- 생성된 홍보 문안의 자동 게시 또는 자동 발송

- "초안이 완성되었습니다" 알람 메일 보내기(추후구현예정)
- 실제 부서 조직도에 따른 담당자 자동 배정

> [!NOTE]
> 실제 부서 조직도 저거는 괜찮은 아이디어일 듯, 지금은 아니지만 추후 구현 예정

- 장기간의 메일 보관 및 검색

## 10. 가상 샘플 데이터

### 가상 수신 메일

```text
발신자: 미래직업교육원 <notice@example.invalid>
수신자: promotion@example.invalid
제목: 2026 여름 데이터 분석 직무교육 참가자 모집

본문:
미래직업교육원에서 대학생을 대상으로 데이터 분석 직무교육
참가자를 모집합니다.

모집 대상: 데이터 분석 직무에 관심 있는 대학생
교육 기간: 2026년 8월 10일~8월 14일
신청 마감: 2026년 7월 31일
주요 혜택: 교육비 전액 지원, 수료증 발급
신청 방법: 예시 신청 페이지에서 온라인 신청
```

### 예상 분류 결과

```text
유형: 직업훈련
대상: 데이터 분석 직무에 관심 있는 대학생
마감일: 2026년 7월 31일
혜택: 교육비 전액 지원, 수료증 발급
신청 방법: 온라인 신청
검토 필요: 신청 페이지 주소와 문의처가 가상 정보이므로 게시 전 확인 필요
```

### 예상 홍보 문안 초안

```text
[데이터 분석 직무교육 참가자 모집]

데이터 분석 직무에 관심 있는 대학생을 대상으로 직무교육이
진행됩니다.

- 교육 기간: 2026년 8월 10일~8월 14일
- 신청 마감: 2026년 7월 31일
- 주요 혜택: 교육비 전액 지원 및 수료증 발급
- 신청 방법: 온라인 신청

상세 신청 주소와 문의처는 담당자 확인 후 게시해 주세요.
```

## 11. 확인할 가설과 부서 질문

- **검증 가설:**
    - 메일 분류와 핵심 정보 추출 결과가 담당자의 초기 확인 시간을 줄인다.
    - 원본과 초안을 함께 보여주면 담당자가 AI 결과를 빠르게 검토할 수 있다.
    - 초안이 완벽하지 않더라도 처음부터 작성하는 것보다 수정하는 편이 효율적이다.
- **문제 이해를 확인할 질문:** 담당자가 현재 메일을 홍보 문안으로 바꾸면서 가장 많은 시간을 쓰는 단계는 무엇인가요?
- **실제 업무 기준을 확인할 질문:** 메일 분류 기준과 홍보 문안에서 반드시 포함해야 하는 항목은 무엇인가요?
- **사용하지 않을 이유를 확인할 질문:** 어떤 오류나 불편이 생기면 담당자가 이 도구를 사용하지 않고 기존 방식으로 돌아갈까요?

## 리스크 및 검증 조건

- 이메일 본문이 Codex CLI에 전달되므로 운영 단계에서는 개인정보, 접근 권한, 저장 범위를 별도로 설계해야 한다.
- 외부 메일에 포함된 지시문이 AI 처리에 영향을 주는 프롬프트 인젝션 위험이 있다. 메일 본문을 명령이 아닌 분석 대상 데이터로 격리해야 한다.
    - 그냥 지금은 지시는 시스템 프롬프트, 본문은 user role로 구분하는 걸로 충분, 격리는 모델의 정렬에 떠넘기기, 탈옥 들어가면 알 바 아님
- 현재 실제 부서의 분류 기준과 홍보 양식은 확인되지 않았다. 해당 기준이 확정되기 전까지 출력 형식은 팀의 가정이다.
